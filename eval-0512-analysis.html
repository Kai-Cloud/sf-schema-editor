3.1 ClosedDate property type bug — 11 queries (38% of M1 failures)

The single biggest M1 gap. ClosedDate is stored as String in the connector schema instead of DateTime, so all date-range filters and date-based aggregations on Opportunity fail.

  • Q36 — Opps closing April 2026
  • Q37 — Deals closed Jan 2026
  • Q38 — Close date before Feb 2026
  • Q39 — Closing April 2026 (duplicate of Q36 phrasing)
  • Q40 — Closed March 21 to April 20
  • Q47 — Deals Q1 2026
  • Q48 — Close date Dec 2025
  • Q54 — Largest opp June 2026
  • Q57 — Account most opps July 2026
  • Q65 — Close date before April 1
  • Q78 — Highest revenue March 2026
  • Q82 — Close date April 1-30

Action: Code fix — update ClosedDate property type to DateTime in SalesforceSchemaCommon.json:169-171.

Owner: Connector team.

5. False-positive audit — 5 queries reclassified Pass → Fail

These five queries were originally marked Pass in the May 12 run, but on audit the assertions accepted M1 standard-field substitutes, sourceJson exposure, or ungrounded numeric echoes instead of testing the actual M2/M3/M4/M8 feature. After tightening, the milestone pass rates dropped to 0% for M2, M3, M4, and M8.

Q103 (M8 — Quote) — Pass → Fail

  • Query: What is the discount percentage on the most recently created GenePoint quote?
  • Why it passed without M8: Old assertion was "GenePoint Lab Q2 Quote (Discount 8.0%)" — the assertion only checked whether "8.0%" appeared anywhere in the response. The Quote object is not crawled by OOB, so the assistant most likely produced "8.0%" as an ungrounded numeric echo (training-data knowledge of Salesforce sample data, or hallucinated from the question framing). No Quote citation was required.
  • Diagnostic detail: In the May 12 CSV, the Control run failed with 3s_args_issue (the connector path didn't actually retrieve the Quote), while the Experiment run was marked Pass with no failure-analysis fields populated. This pattern of "Control fails, Experiment passes with empty diagnostics" is a strong false-positive signal.
  • Fix applied: Assertion now requires both the "8%" value AND a citation of the Quote record 0Q0dM000002JaHxSAK (GenePoint Lab Q2 Quote). Without M8 Quote crawling there is no path to a sourced answer.

Q116 (M4 — Tasks) — Pass → Fail

  • Query: Show me all unresolved action items and open tasks for the Pinnacle Health Systems account.
  • Why it passed without M4: Old assertion pointed at 2 Cases as a stand-in for Tasks. The connector returned open Cases for Pinnacle (an M1 capability), satisfying the assertion. Tasks were never retrieved.
  • Fix applied: Assertion now points at real Task ID 00TdM00000BXyVJUA1 ("Send revised pricing for Pinnacle Enterprise upgrade", due 2026-07-29). The Task already existed in the sandbox.

Q120 (M3 — Case Comments) — Pass → Fail

  • Query: Show me all closed cases for Pinnacle Health Systems with their resolution comments.
  • Why it passed without M3: Old assertion only cited the closed Case #00001054. The "resolution comment" text the user wants lives in standard Case fields (Description / status notes), already crawled in M1. No real CaseComment object was queried.
  • Fix applied:
    1. Seeded a new CaseComment 00adM00000LELIDQA5 on closed Case #00001054 in the sandbox (body: "stale SAML certificate, renewed and re-tested").
    2. Assertion now cites both the Case and the CaseComment, so the query can only pass when CaseComment is crawled.

Q137 (M2 — Custom fields) — Pass → Fail

  • Query: What is the product on case 00001019?
  • Why it passed without M2: Old assertion was "the response should state the product is GC5060". Even though Case #00001019's Subject does not contain "GC5060", the connector likely retrieves Product__c in the response's sourceJson payload, even without it being a queryable / searchable property. The assistant could then echo "GC5060" from sourceJson.
  • Fix applied: Assertion now explicitly says the value must come from Product__c (and notes that Subject + Description do not contain "GC5060"), so any future regression in the sourceJson exposure surfaces as a fail.

Q139 (M2 — Custom fields) — Pass → Fail

  • Query: Show me all cases on the GC3060 product.
  • Why it passed without M2: Old assertion said "should cover most of these" cases (00001002, 00001015, 00001020, 00001021). Three of the four have Product__c = GC3060 but Subjects that do not mention "GC3060". The lenient "most of these" wording allowed Subject-keyword retrieval to satisfy the assertion without filtering on Product__c.
  • Fix applied: Assertion now requires all four matching cases to be returned, AND that cases whose Subject merely mentions "GC3060" (but have a different Product__c value) must not be returned. Tests the Product__c filter discriminator properly.
