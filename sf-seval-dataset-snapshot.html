2. The 2026-05-09 date effect #

All seeding happened on 2026-05-09. Salesforce auto-stamps several read-only date fields with the seeding-run timestamp. Some fields the narrative describes as 5/18, 6/12, 6/23, etc. actually all collapsed to 2026-05-09 in the org. This is the most important fact for prompt authors to internalize.

⚠ Read-only date fields clustered on 2026-05-09
Any prompt that filters on these fields with a window overlapping 5/9 will match ~1,900 records and bust the 50-record citation cap.
  • CreatedDate — every Layer 3 + Layer 4 record (~1,900)
  • LastModifiedDate — same as above
  • SystemModstamp — same as above
  • Lead.ConvertedDate — all 47 truly-converted Layer 4 Leads + 5 Layer 3 conversions
  • Case.ClosedDate — all ~110 Layer 4 closed Cases + 7 Layer 3 closed Cases
✓ Writable date fields seeded per narrative (safe to filter on)
These were set explicitly by the seed scripts to match the simulation timeline. Distributed across 2026-05 to 2026-10.
  • Opportunity.CloseDate — 7 closed-won spread Apr–Jul 2026; 12 open Jul–Oct 2026
  • Task.ActivityDate — Layer 3 only (1 record); distributed per narrative
  • Event.StartDateTime / EndDateTime — Layer 3 only (1 record)
  • Campaign.StartDate / EndDate — May 15 – Sep 30 2026
  • Order.EffectiveDate / EndDate — Layer 3 only

4. Owner distribution — threshold-design targets #

The three real SF Admin users are deliberately assigned to probe three platform thresholds: ~1000 (Glean's per-object API ceiling), ~50 (Microsoft's 50-record RAG citation cap), and ~1–3 (minimum owner-filter sentinel). Owner-filtered prompts must use these three names.

Object Kai bulk Maple ≤ 50 Nitin ≤ 3 Threshold hit? Notes
Account 69 28 2 ✓ Maple short — L4 noise concentrated to Kai
Contact 183 32 2 ✓ LeadConvert auto-creates inherit Lead owner → all Kai
Lead 1,001 11 1 ✓ Kai > 1000-ceiling; Nitin = sentinel
Opportunity 369 50 3 ✓ Maple = exact 50-cap
Case 112 48 3 ✓ 8 Cases reassigned from Maple → Kai on 2026-05-11 to bring Maple under 50
ℹ Why Account / Contact Maple counts are under target
The LeadConvert Step-3c S4 strategy auto-assigns each created Contact to the converting Lead's existing owner (Kai). Owner re-balance was deliberately skipped — we accepted the asymmetry to avoid post-conversion mutations. Future iterations needing balanced Contact ownership must run a separate UPDATE Contact SET OwnerId=... pass after LeadConvert.
5. Prompt-authoring rules derived from the dataset #

Hard rules for the SEVAL queryset (both cape-collect-glean-parity-queryset.csv and seval-guided-queryset.csv). Source: simulation doc §7 and demo-data/data-plan.md §7.

RuleAvoidUse instead
7.3.a Do not filter on auto-stamped dates with a window overlapping 5/9 "leads converted on May 9 2026" (returns ~52) "leads with status Closed - Converted"
7.3.b Use writable date fields when a date filter is needed CreatedDate, LastModifiedDate, ConvertedDate, ClosedDate Opportunity.CloseDate, Task.ActivityDate, Event.StartDateTime, Campaign.StartDate
7.3.c Reuse the existing absolute-date phrasings from the 111-set relative dates ("past 7 days") run shortly after 5/9 "Q1 2026", "April 2026", "between March 21 and April 20"
7.3.d Aggregation prompts bound by writable date, not auto-stamped "largest opp created in May 2026" "largest opp closed in May 2026"
7.3.e Treat 5/9 as "do-not-touch" in prompt phrasing "on May 9", "this week" (run shortly after 5/9), "in the second week of May" Use later distributed dates (5/12, 5/18, 5/26, 6/3, etc.)
Owner Filter by OwnerId only via Kai / Maple / Nitin "Ben's open opportunities", "deals owned by Alice" "Kai Liang's open opportunities" — Ben/Alice surface via full-text search
⚠ Symptom of an accidental 5/9 filter
If a SEVAL prompt unexpectedly returns ~1,900 records (or 5 / 47 / 110 for ConvertedDate / ClosedDate), the prompt is filtering on an auto-stamped date that clusters on 2026-05-09. This is a queryset bug, not a connector bug.
