FLS tradeoff: Both connectors have FLS limitations, but in opposite directions. Glean over-exposes fields (indexed but not filtered at query time). Microsoft under-indexes them (blocks entire field if FLS is set on any profile). Neither approach is ideal. FCC sidesteps this entirely because it uses the user's own Salesforce session, so native FLS applies.
⚠ Glean has broader ACL coverage (Territory Management, hourly ACL full crawl of all share tables). Microsoft has a known bug with Lead defaults. Both have FLS limitations. FCC resolves FLS gaps by using native Salesforce permissions at query time, but hourly ACL crawl requires platform-level support.

Bottom line: Glean's Salesforce connector is substantially more capable, particularly for organizations needing CRM data beyond the 5 core objects, file/attachment search, territory-based permissions, or near-real-time freshness.

Closure strategy:
• OOB: add after MVP High-priority objects (Tasks, Events, Case Comments, Files) planned for the OOB connector post-MVP.
• OOB: leverage custom Remaining objects (Campaigns, Products, Orders, CPQ, Custom Objects, etc.) are already supported by custom connector implementations. OOB engineering should reference that work to accelerate delivery.
• FCC (Federated Content Connector) closes the real-time, FLS, and file gaps by querying Salesforce APIs directly with user credentials at query time. This is the architectural equivalent of Glean's Live Mode.
• Platform Hourly ACL full crawl (scanning Salesforce share tables like AccountShare, CaseShare to keep permissions current) requires platform-level infrastructure. Salesforce does not expose deleted share records via incremental APIs, so a dedicated full scan is needed. Neither OOB nor custom connector can implement this without platform support.
